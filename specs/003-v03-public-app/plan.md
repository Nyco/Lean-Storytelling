# Implementation Plan: Lean Storytelling v0.3 — Public Web App

**Branch**: `003-v03-public-app` | **Date**: 2026-03-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-v03-public-app/spec.md`

---

## Summary

Migrate the Lean Storytelling app from a local static app to a publicly deployable full-stack web application. The backend is Node.js + Fastify serving a REST API and static frontend files. Data is persisted in PostgreSQL. Authentication is passwordless via magic link (email). Deployment uses Docker Compose for both local dev and production VPS. New features include account management, Stories List, and multi-version story management.

---

## Technical Context

**Language/Version**: Node.js 20 LTS (backend) + Vanilla JS ES2020 (frontend, unchanged from v0.2)
**Primary Dependencies**: Fastify 5, `@fastify/jwt`, `@fastify/cookie`, `@fastify/static`, `@fastify/cors`, `postgres` (pg client), `resend`, `crypto` (Node built-in for token hashing)
**Storage**: PostgreSQL 16 (server-side persistence); browser sessionStorage (anonymous session, existing)
**Testing**: Manual acceptance testing for v0.3 user-testing phase; no automated test framework added
**Target Platform**: Linux VPS (production), Ubuntu/macOS (local dev), Docker Compose
**Project Type**: Full-stack web application (REST API + static frontend)
**Performance Goals**: Response time < 500ms for all API endpoints under single-user testing load
**Constraints**: Zero external CDN at runtime; offline-capable frontend (existing service worker); large-screen only (mobile deferred); SameSite=Lax cookies for magic link redirect compatibility
**Scale/Scope**: Single-user to small-group user testing (~10–50 users); single VPS (1 CPU / 2 GB RAM)

├── package.json
└── Dockerfile

frontend/
├── index.html
├── style.css
├── app.js                   # existing v0.2 app, extended with auth + sidebar
├── api-client.js            # NEW: thin wrapper around fetch for API calls
├── auth.js                  # NEW: magic link flow, session management, sessionStorage migration
├── sidebar.js               # NEW: Stories List + Versioning UI
├── auto-save.js             # NEW: debounced save + offline queue

**Structure Decision**: Web application (Option 2). Frontend and backend are separate directories. Frontend is served as static files by Fastify (`@fastify/static`) — no separate nginx container needed for v0.3.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| §III Mobile support deferred | v0.3 is user-testing phase on large-screen devices; mobile layout would double CSS work and is not required for the test cohort | Doing it now would delay the user-testing deployment with significant CSS effort for a non-essential audience segment; tracked for v0.4 |
| Backend added (breaks "zero runtime dependencies for v1") | Server-side persistence, auth, and multi-device access require a backend | localStorage/sessionStorage cannot provide cross-device persistence or account-based storage; this is the explicit goal of v0.3 |
